# SECURITY.md — معمارية الأمان الكاملة لـ AdLoop

**آخر تحديث:** 18 يوليو 2026
**الحالة:** وثيقة حية — أي ميزة جديدة تؤثر على الأمان لازم تُحدّث هنا قبل
ما تُعتبر مكتملة. مهندس أمان جديد ينضم للمشروع لازم يقدر يفهم المعمارية
الكاملة من الملف ده بس، من غير الرجوع لأي محادثة سابقة.

---

## 1) فلسفة الأمان

**المبدأ الأساسي: Zero Trust مطبّق بواقعية، مش بحجم Google.**

منتج AdLoop بيتعامل مع أصول حساسة جداً: توكنات وصول مباشر لحسابات
إعلانات حقيقية (جوجل/ميتا)، بيانات عملاء (أرقام تليفونات، محادثات)،
وبيانات مالية مستقبلية. المبدأ اللي بنشتغل بيه:

> **متثقش في حاجة افتراضياً. تحقق من كل طلب. افترض إن الاختراق ممكن
> يحصل، وصمّم عشان الضرر يكون محدود لو حصل.**

**لكن — وعي بالتكلفة صريح:** إحنا شركة ناشئة قبل أول عميل دافع، مش Google.
كل قرار أمني هنا موزون بين "الحماية الحقيقية اللي محتاجينها دلوقتي"
و"البنية المؤسسية اللي مش هتفيدنا إلا بعد سنين من النمو". القسم الأخير
من الملف ده (الخارطة المستقبلية) بيوضح إيه اللي اتأجّل، وليه بالتحديد.

## 2) نموذج التهديد (Threat Model)

**مين ممكن يهاجمنا، وبيحاول يوصل لإيه:**

| المهاجم | الهدف | مثال |
|---|---|---|
| مستخدم عادي خبيث | بيانات مستخدم تاني | تخمين ID مورد (BOLA) |
| بوت آلي | حسابات وهمية، إغراق endpoints عامة | تسجيل حسابات جماعي، إغراق تتبع CTA |
| مخترق خارجي | قاعدة البيانات، السيرفر | استغلال ثغرة في مكتبة، تسريب بيانات اتصال |
| موظف/مقاول سابق | كود المصدر، أسرار | وصول لمستودع GitHub بعد انتهاء التعاون |
| منافس | نسخ آلية المنتج | Reverse Engineering للواجهة العامة (غير قابل للمنع بالكامل) |
| جهة تنظيمية | عدم امتثال قانوني | قانون حماية البيانات المصري (تفصيل في قسم 15) |

**الأصول الأخطر (لو اتسرقت، الضرر الأكبر):**
1. توكنات OAuth (وصول مباشر لحسابات إعلانات عملاء حقيقيين)
2. بيانات المصادقة (كلمات سر، جلسات)
3. بيانات التواصل مع العملاء (أرقام تليفونات، محادثات واتساب)
4. مفتاح تشفير التوكنات نفسه (`TOKEN_ENCRYPTION_KEY`)

## 3) حدود الثقة (Trust Boundaries)

```
[متصفح المستخدم]  <-- حد ثقة -->  [Next.js API Routes]  <-- حد ثقة -->  [قاعدة البيانات]
                                        |
                              <-- حد ثقة -->  [Google/Meta APIs الخارجية]
                                        |
                              <-- حد ثقة -->  [Webhooks خارجية (سلة، واتساب)]
```

**كل عبور لحد ثقة بيتطلب تحقق صريح:**
- متصفح إلى API: جلسة JWT صالحة + (لو endpoint عام) حد استخدام
- API إلى قاعدة البيانات: فحص ملكية إجباري لكل مورد (مش دخول مباشر بـ ID)
- API إلى منصات خارجية: توكن مُفك التشفير وقت الاستخدام بس، مش مُخزّن مفكوك أبداً
- منصات خارجية إلى API (Webhooks): توقيع HMAC مُتحقق منه قبل أي معالجة

## 4) استراتيجية المصادقة (Authentication)

**الطريقة:** JWT في كوكي HttpOnly، مش localStorage (يمنع سرقة التوكن عن
طريق XSS حتى لو حصلت).

| الخاصية | القيمة | السبب |
|---|---|---|
| تخزين الجلسة | كوكي httpOnly, secure, sameSite=lax | يمنع الوصول من JavaScript، ويمنع CSRF جزئياً |
| مدة الجلسة | 30 يوم | توازن بين الراحة والأمان لمنتج B2B مش بنكي |
| تشفير كلمة السر | bcrypt، 12 rounds | معيار صناعي، مقاوم لهجمات القوة الغاشمة على الهاش نفسه |
| قوة كلمة السر | 8 أحرف+ حرف كبير+ حرف صغير+ رقم+ رمز، مؤشر حي في الواجهة | يمنع الرفض المربك بعد الإرسال، ونفس السياسة مطبّقة وقت الاسترجاع كمان (مش أضعف) |
| قفل بعد محاولات فاشلة | 5 محاولات ثم قفل 15 دقيقة | يوقف التخمين الآلي من غير ما يحبس مستخدم نسي كلمة السر للأبد |
| استرجاع كلمة السر | توكن عشوائي 32 بايت، صالح ساعة واحدة | أقصر من تحقق البريد (24 ساعة) لأنه بيدّي تحكم كامل في الحساب |
| تحقق البريد الإلكتروني | إجباري بعد التسجيل، غير مانع للدخول | نوازن بين التحقق والاحتكاك، نوري تنبيه بدل ما نمنع الاستخدام |
| الكابتشا | Cloudflare Turnstile عند التسجيل | يمنع التسجيل الآلي الجماعي |

**بدائل اتفكر فيها ورُفضت (دلوقتي):**
- **MFA إجباري:** رفضناه دلوقتي — احتكاك عالي لعدد مستخدمين صغير جداً،
  والقيمة الأمنية الحدّية منخفضة قبل ما يكون فيه بيانات دفع فعلية.
  مسار الترقية: جاهز نضيفه كخيار اختياري (TOTP) وقت ما العدد يكبر
  أو الدفع يتفعّل، مفيش تغيير معماري كبير مطلوب.
- **JWT قصير + Refresh Token دوّار:** رفضناه دلوقتي لبساطة الحجم الحالي.
  مسار الترقية: لو حبينا نقدر نُبطل جلسة معينة فوراً (مش بس الانتظار
  لحد انتهاء الـ 30 يوم)، محتاجين قائمة إبطال (Denylist) في قاعدة
  البيانات — تغيير متوسط الحجم، مش إعادة بناء.

**فجوة معروفة (موثّقة بصراحة):** مفيش إبطال جلسات نشطة عند تغيير كلمة
السر حالياً (لأن مفيش endpoint لتغيير كلمة السر من داخل الحساب أصلاً
لسه، بس استرجاعها من برة موجود). مسجّل في قسم 20 كبند تالي.

## 5) استراتيجية التفويض (Authorization)

**المبدأ: كل مورد بياخد id لازم يتحقق من الملكية قبل أي عملية، بدون استثناء.**

النمط الموحّد المستخدم في كل الكود:
```typescript
const resource = await prisma.model.findFirst({
  where: { id: params.id, workspace: { userId: user.id } },
});
if (!resource) return NextResponse.json({ error: "not found" }, { status: 404 });
```

**ملاحظة مهمة:** بنرجّع 404 مش 403 لو المورد مش موجود أو مش بتاع
المستخدم، عشان منأكدش لمهاجم إن الـ ID نفسه موجود أصلاً (Information
Disclosure بسيطة، لكن مفيدة).

**مراجعة شاملة تمت في 18 يوليو 2026** لكل route فيه id (فحص فعلي
لكل ملف، مش بحث نصي) — لقيت وأصلحت 4 ثغرات BOLA حقيقية (تفصيل كامل في
docs/security-audit-2026-07-18.md).

**تصعيد الصلاحيات الرأسي (Vertical Privilege Escalation):** حقل
isAdmin مفيهوش أي مسار API لتفعيله — تحديث يدوي مباشر في قاعدة
البيانات فقط. قرار متعمد: زرار "خليني أدمن" في الواجهة، حتى لو محمي،
بيبقى نقطة هجوم إضافية.

## 6) إدارة الجلسات (Session Management)

مُغطّى في قسم 4. إضافة واحدة: **تقمّص الحساب (Impersonate)** في لوحة
الأدمن بيستخدم نفس آلية الجلسات (JWT جديد للمستخدم المستهدف)، لكن بيحتفظ
بهوية الأدمن الأصلي في كوكي منفصلة (impersonating_by) عشان الرجوع
يكون ممكن من غير تسجيل دخول تاني. كل عملية تقمّص مسجّلة إجبارياً في
AdminAuditLog — قوة بالحجم ده لازم تيجي مع تسجيل، مش اختياري.

## 7) إدارة التوكنات (Token Management)

| نوع التوكن | التخزين | التشفير | الدورة الزمنية |
|---|---|---|---|
| جلسة المستخدم (JWT) | كوكي HttpOnly | موقّع فقط (لا بيانات حساسة، فقط userId) | 30 يوم، بدون تجديد تلقائي |
| Google Ads Refresh Token | قاعدة البيانات | AES-256-GCM | يفضل صالح لحد ما يُلغى يدوياً |
| Meta Access Token (Long-Lived) | قاعدة البيانات | AES-256-GCM | 60 يوم، تجديد يدوي مطلوب من العميل |
| مفتاح التشفير نفسه | متغير بيئة منفصل | — | ثابت، لو اتغيّر كل التوكنات المشفّرة القديمة تفقد صلاحيتها |

**قرار مهم موثّق:** فكّرنا في الانتقال لـ Meta System User Tokens (بلا
انتهاء) بدل الـ 60 يوم الحالية، لكن اكتشفنا إن ده يتطلب Business
Integration System User (BISU) لمنصات SaaS متعددة العملاء زينا — محتاج
تنفيذ "Embedded Signup" ومراجعة Meta Tech Provider إضافية. ده تكامل
منفصل كبير، مش تعديل كود — مؤجّل بوعي، والبديل المؤقت (تنبيه استباقي
قبل انتهاء التوكن بأسبوع) مبني وشغّال.

## 8) إدارة الأسرار (Secret Management)

**الحالة الحالية:** متغيرات بيئة (Environment Variables) في Vercel،
مشفّرة أثناء الراحة من عند Vercel نفسها. مفيش أسرار في الكود المصدري
أبداً — .env.example دايماً فاضي من القيم الحقيقية.

**ليه مش Secrets Manager مخصص دلوقتي؟** توثيق جوجل الرسمي بيوصي بيه
للمشاريع الكبيرة، لكن لحجمنا الحالي (تطبيق واحد، مستخدم واحد لغاية
دلوقتي)، متغيرات بيئة Vercel المشفّرة كافية. مسار الترقية: لو الفريق
كبر لأكتر من شخص أو اتعدد البيئات (staging/prod منفصلة فعلياً)، وقتها
الانتقال لـ Secrets Manager مخصص يستاهل — تغيير سطحي نسبياً.

## 9) أمان الـ APIs

**التحقق من المدخلات (Input Validation):** حالياً على مستوى TypeScript
وقت الترجمة بس، مش فحص وقت التشغيل (Runtime) للشكل الفعلي للبيانات
الواردة. ده فجوة حقيقية معترف بيها — JSON body بيتفكك مباشرة من غير
مكتبة تحقق (زي Zod). مسجّل كبند أولوية عالية في قسم 20.

**CSRF:** الجلسة معتمدة على كوكي (مش Bearer token في الهيدر)، فـ CSRF
تهديد نظري حقيقي. الحماية الحالية: sameSite=lax على الكوكي — بيمنع
إرسال الكوكي في أغلب طلبات POST من موقع تاني، لكن مش حماية كاملة 100%.
مسار الترقية: توكن CSRF صريح (Double Submit Cookie Pattern) لو حسّينا
إن الخطر زاد (مثلاً بعد إضافة الدفع).

**Rate Limiting:** مبني لـ endpoints معينة عالية الخطورة (تسجيل، تسجيل
دخول، تتبع CTA العام) عن طريق lib/rateLimit.ts (نافذة زمنية ثابتة،
مخزّنة في قاعدة البيانات). مش شامل كل الـ APIs — قرار واعٍ: الأولوية
للـ endpoints المفتوحة (بدون تسجيل دخول) لأنها الأخطر.

**Webhook Security:** كل الـ webhooks الواردة (سلة، واتساب) محمية
بتوقيع HMAC، ومحمية من إعادة التشغيل (Replay) عن طريق
ProcessedWebhookEvent (معرف فريد لكل حدث، مش هيتنفذ مرتين).

**حدود حجم الطلب:** Next.js/Vercel عندهم حد افتراضي معقول. رفع الملفات
(upload-sheet) له حد صريح إضافي (5 ميجابايت) وفحص نوع الملف قبل
المعالجة — يمنع إغراق الموارد بملف ضخم أو غير صالح.

## 10) أمان قاعدة البيانات

**التشفير أثناء الراحة:** معتمد على مزوّد الاستضافة (Supabase/Neon
بيوفروا تشفير قاعدة البيانات كاملة افتراضياً) — لازم يتأكد صراحة وقت
اختيار المزوّد النهائي، مسجّل في activation-checklist.md.

**بيانات حساسة إضافية غير مشفّرة على مستوى العمود:** أرقام التليفونات
(في SessionConversion)، أسماء العملاء — معتمدين على تشفير قاعدة
البيانات الكامل (مش تشفير عمود بعمود). قرار واعٍ: تشفير كل عمود لوحده
تعقيد كبير (فهرسة، بحث، أداء) لقيمة أمنية إضافية محدودة لو تشفير القرص
كامل موجود أصلاً.

**الحذف (Delete Strategy):** حالياً حذف نهائي (Hard Delete) لمعظم
العمليات (حذف مساحة عمل، حذف حساب) — مش Soft Delete. السبب: حق "المحو"
(Right to Erasure) في قانون حماية البيانات بيتطلب حذف فعلي في النهاية
على أي حال، والاحتفاظ ببيانات "محذوفة" فترة أطول بيزوّد المخاطر
القانونية مش بيقللها. التعويض عن فقدان إمكانية التراجع: تأكيد صريح
مطلوب قبل أي حذف (كتابة الاسم/كلمة السر)، مش زرار واحد.

**النسخ الاحتياطي:** معتمد على مزوّد الاستضافة حالياً — لازم يتأكد
تفعيل النسخ الاحتياطي التلقائي والتشفير بتاعه صراحة وقت الإطلاق الفعلي.

## 11) استراتيجية التشفير

| البيانات | الطريقة | لماذا |
|---|---|---|
| كلمات السر | bcrypt (تشفير أحادي الاتجاه) | لا يمكن فك التشفير أبداً، حتى لنا |

| قاعدة البيانات كاملة | تشفير المزوّد أثناء الراحة | آخر طبقة دفاع لو حصل وصول فعلي للقرص |

## 12) النسخ الاحتياطي والاستعادة

**الحالة الحالية:** معتمدة بالكامل على مزوّد الاستضافة. فجوة حقيقية
معترف بيها: مفيش اختبار استعادة (Restore Drill) دوري، ومفيش نسخة
احتياطية منفصلة عن مزوّد الاستضافة الأساسي.

**مسار الترقية:** نسخة احتياطية أسبوعية آلية لملف خارجي، مشفّرة، مع
اختبار استعادة كل ربع سنة على الأقل. مش أولوية اليوم لعدم وجود بيانات
عملاء حقيقية بعد، لكن أولوية فورية أول ما أول عميل حقيقي يدخل بياناته.

## 13) التسجيل والتدقيق (Logging & Auditing)

- أخطاء النظام: Sentry (مجاني، Developer tier)
- أفعال الأدمن: AdminAuditLog — كل فعل قوي (تقمّص، تعليق حساب) مسجّل إجبارياً
- صحة المزامنة اليومية: CronRunLog — نجاح/فشل كل تشغيل، بالتفصيل
- إخفاء البيانات الحساسة من السجلات: مش مُطبّق بشكل منهجي حالياً — لازم
  مراجعة يدوية دورية للتأكد إن console.error أو Sentry مبيسجلوش توكنات
  أو كلمات سر بالغلط. بند أولوية متوسطة، قسم 20.

## 14) المراقبة والتنبيه (Monitoring & Alerting)

- كشف بوتات (تكرار IP، بصمة User Agent، كليكات بدون تحويل) — lib/qualitySignals.ts
- كشف محاولات دخول فاشلة متكررة — قفل تلقائي بعد 5 محاولات
- تنبيه استباقي لانتهاء توكنات المنصات — قبل الانتهاء بأسبوع
- مفيش نظام كشف شذوذ على مستوى استخدام حساب المستخدم نفسه (زي دخول من
  دولة غريبة فجأة) — مؤجّل بوعي، قيمته الحقيقية بتزيد مع عدد المستخدمين.

## 15) الخصوصية — ذات صلة مباشرة بقانون حماية البيانات المصري

**عاجل: موعد نهائي قانوني 31 أكتوبر 2026.** اللائحة التنفيذية لقانون
151/2020 صدرت نوفمبر 2025 وبقى القانون شغّال بالكامل. المتطلبات
الأساسية (تفصيل كامل في activation-checklist.md قسم 14):
- ترخيص من مركز حماية البيانات الشخصية (PDPC)
- ترخيص منفصل لنقل البيانات عبر الحدود (بياناتنا خارج مصر)
- تعيين مسؤول حماية بيانات (DPO) في حالات معينة
- إخطار إجباري خلال 72 ساعة لأي اختراق بيانات

**اتبنى في المراجعة دي (18 يوليو):**
- حق المحو (Right to Erasure) — حذف حساب كامل
- حق قابلية النقل (Data Portability) — تصدير بيانات JSON

**لسه ناقص:**
- سياسة احتفاظ بيانات صريحة (كام يوم/شهر نحتفظ بـ MetricSnapshot،
  CtaClickEvent قبل ما نمسحها تلقائياً؟) — حالياً بلا حد زمني
- سياسة تقليل البيانات (Data Minimization) — مراجعة صريحة لكل حقل بنجمعه
- **📌 بانر موافقة الكوكيز (Cookie Consent)** — طلب صريح من المستخدم
  ننفّذه بعد ما نخلص موضوع الاختبار العدائي/الأمان بس، مش دلوقتي.
  مسجّل أيضاً في `activation-checklist.md` قسم 14.

## 16) أمان الدفع (لسه مش مبني)

**الحالة:** صفر كود. لما يُبنى، المبادئ الإلزامية:
- مفتاح Idempotency لكل محاولة شحن (يمنع الشحن المزدوج لو الطلب اتكرر)
- توقيع Webhook من مزوّد الدفع (Kashier/PayMob) مُتحقق منه، نفس نمط Webhooks الحالية
- مطابقة (Reconciliation) دورية بين سجلاتنا وسجلات مزوّد الدفع

## 18) أمان المهام الخلفية (Background Jobs)

**النمط الحالي:** Vercel Cron مع after() للتنفيذ في الخلفية بعد الرد.
مش قائمة انتظار حقيقية (Queue) — لا Dead Letter Queue، لا قفل موزّع.
قرار واعٍ: لحجم الاستخدام الحالي (تشغيل يومي واحد)، احتمال التشغيل
المتزامن ضئيل جداً وVercel Cron نفسها بتضمن عدم التكرار. مسار الترقية:
لو زاد التكرار أو الحجم، قائمة انتظار حقيقية بديل مباشر.

**عزل الأخطاء:** كل Workspace بيتعالج في حلقة منفصلة — فشل واحد مبيوقفش الباقي.

## 19) حماية منطق الأعمال (Business Logic)

- تكرار التحويلات: منع مزدوج (على مستوى الجلسة + رقم الهاتف) — lib/ctaDeduplication.ts
- تحويلات وهمية: كشف بوتات + كشف "ضغطات بالخطأ" على ماسنجر
- إساءة استخدام الأتمتة: سقف أقصى لكل قفزة، فترة تهدئة، موافقة إجبارية افتراضياً
- إساءة استخدام حصة الذكاء الاصطناعي: الحد على مستوى المستخدم مش الـ Workspace
- تعداد روابط التقارير العامة: توكن عشوائي طويل، لكن مفيش Rate Limiting

## 20) تحديث (نفس اليوم) — من "مؤجّل" إلى "مبني فعلياً"

بعد مراجعة صادقة، اتضح إن معظم البنود "المؤجّلة" كانت قرار أولوية مني،
مش عائق حقيقي. اتبنوا كلهم في نفس الجلسة:

| البند | الحالة الجديدة |
|---|---|
| Runtime input validation (Zod) | ✅ مبني - `lib/validation/schemas.ts`، مطبّق على 8+ من أخطر الـ routes (مصادقة، أفعال أدمن، تسعير) |
| CSRF token صريح | ✅ مبني - نمط Double Submit Cookie (`lib/csrf.ts`)، مفعّل على أخطر 3 أفعال (تقمّص، تعليق حساب، حذف حساب) |
| MFA | ✅ مبني بالكامل - TOTP (`lib/mfa.ts`)، تدفق دخول بخطوتين، سر مشفّر AES-256-GCM |
| سياسة احتفاظ بالبيانات | ✅ مبني - `lib/dataRetention.ts`، تنظيف تلقائي يومي للبيانات الخام منتهية الصلاحية |
| security.txt | ✅ مبني - `public/.well-known/security.txt` (RFC 9116) |
| نسخ احتياطي مستقل | ✅ مبني - `lib/backup.ts`، Vercel Blob (مزوّد مختلف تماماً عن قاعدة البيانات)، أسبوعي، آخر 8 نسخ محفوظة |

**الباقي فعلاً مؤجّل بعائق خارجي حقيقي (مش قرار أولوية):**

| الفجوة | ليه مؤجّلة فعلياً | متى تستاهل |
|---|---|---|
| Soft Delete | قرار واعٍ (مش عائق) - الحذف النهائي أنسب لمتطلب "حق المحو" القانوني | لو ظهرت حاجة فعلية لاسترجاع بيانات محذوفة بالغلط |
| Secrets Manager مخصص | محتاج قرار بنية تحتية ومصاريف إضافية | لو الفريق كبر أو تعددت البيئات |
| BISU (Meta System User) | عائق فعلي - محتاج موافقة Meta خارجية | لو مشكلة انتهاء التوكن كل 60 يوم بقت مزعجة فعلياً |
| قائمة انتظار مهام حقيقية | الحجم الحالي مايستاهلش تعقيد إضافي | لو تكرار المزامنة زاد عن مرة يومياً |
| Zod على كل route (تغطية كاملة) | اتبنى على أخطر الـ routes، التوسعة الكاملة عمل مستمر مش نقطة واحدة | تدريجياً مع كل route جديد |
| CSRF على كل route تغييري | نفس المبدأ - اتبنى على الأخطر، التوسعة عمل مستمر | تدريجياً

## 21) خطة الاستجابة للحوادث — إطار أولي

5. إخطار العملاء المتأثرين مباشرة

هذا إطار أولي بسيط، مش خطة كاملة معتمدة من محامٍ — لازم مراجعة قانونية.
